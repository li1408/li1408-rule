## 从一张白纸开始搭工作台

最开始，我的目标其实很朴素：把 MiKTeX、VS Code、LaTeX Workshop、Codex、Python、Git 和 GitHub 串起来，搭一个真正能干活的数学建模工作台。

这里面，MathModelHub 被我放在 `resources` 下，只作为只读参考库。它可以提供思路和资料，但不能反过来污染项目结构，更不能让比赛工程变成“从参考仓库里复制一点、再从聊天记录里捞一点”的考古现场。

我先给比赛项目定了一套规范化目录：代码、原始数据、处理后数据、图表、论文、模型结果、运行日志和提交检查各自隔离。原始数据默认只读，大文件和运行产物统一放在 E 盘，不往 C 盘疯狂堆缓存。以前我总觉得“文件放哪儿都一样，反正搜索能找到”，直到某次同时面对好几个 `final_v2_really_final`，才意识到这不是自由，是失控。

## 先把论文编译这件小事做稳定

中文论文采用 `ctexart + XeLaTeX + BibTeX + gbt7714-numeric`。平时可以借助 LaTeX Workshop 快速预览，但正式编译入口统一使用 `latexmk`，辅助文件全部进入 `build`，避免项目根目录被 `.aux`、`.log`、`.toc` 淹没。

真正准备匿名提交时，论文、图表、结果和提交包里都不能出现作者身份。署名可以存在于仓库说明和项目元数据中，但绝不能顺手混进比赛材料。匿名不是最后删掉名字就完事，而是从生成链路上保证身份信息根本进不去。

环境配置当然没有想象中浪漫。MiKTeX 包、XeLaTeX、编辑器插件和路径变量轮流给我上强度。某次看着“编辑器里能编译，命令行里却找不到”的场面，我只想说一句：艹了，原来同一台电脑也能拥有两套互不认识的世界观。

## 从单题脚本升级成通用架构

最早的版本只解决“这一道题如何可复现”。后来我发现，如果核心代码里到处写着某道题的字段名、文件名和特殊判断，那它顶多算一个整理得比较漂亮的项目，算不上工作流。

于是我把整体拆成 `competition_profiles`、`workflow_core`、`plugins`、`cases` 和 `paper_templates`。核心层不能导入任何具体题目；具体比赛通过 profile 和插件接入；矿井题则退回它应有的位置——一个重要的回归案例，而不是整个宇宙的中心。

这次改造最费脑子的不是搬文件，而是辨认哪些逻辑真正通用，哪些只是被我误认为通用。拆到一半发现某个“通用校验器”其实偷偷依赖题目状态时，我又一次感叹：艹了，模型漏洞有时候不是模型不会算，而是代码把前提藏得太深。

## 给每一次运行留下“案发现场”

为了让结果不再依赖“我记得当时大概这么跑的”，工作流加入了 `run_id`、run manifest、artifact manifest、配置快照和输入 hash。关键运行可以冻结，证据文件不可修改，还要支持冷启动复现：换一个干净环境，仅凭记录就能知道用了什么输入、什么配置、生成了哪些产物。

同时，我建立了 H1—H8 人工门禁。自动化负责整理证据、执行检查和报告异常，但关键节点必须有人明确确认。机器不能因为脚本返回 0，就替我宣布“模型正确”。

建模过程也被拆成一组可检查的材料：problem profile 描述问题结构，requirements matrix 对应每一问的要求，assumptions registry 记录假设，model cards 解释模型用途和边界，validation plan 规定如何验证。审计则按风险分为 A0—A3，并加入比较审计、claim/evidence 双表和 figure manifest，让论文里的强结论、证据来源与图表之间可以互相追踪。

P0 质量闭环进一步补上逐问故事板、结果卡、摘要矩阵、插件验证梯度、强结论证据门禁，以及私有优秀论文隔离。参考可以帮助我发现表达方式，但不能让私有材料进入公开仓库、提示词或最终论文。

## 两个 AI 审稿，但最后签字的还是人

审核材料会自动按顺序进入 `09_review_packages`。每个门禁都包含共享载荷、AI 提示词、两份 AI 回复、交叉核对、人工清单和最终人工决定。

双 AI 审核最终采用 Google AI Studio 的 Gemini 3.1 Pro Preview，加一个独立的 Codex 审稿任务。两边看到的是相同 hash 的材料，彼此看不到对方意见，避免互相带节奏；最后是否通过，仍由人批准。

实际测试证明这种独立视角确实有价值。Gemini 把 Q4 中“通知时刻工人位置缺失”列为 WARNING；独立 Codex 又进一步指出题面源证据、Q2 输入、Q3 规则继承以及 Q4 状态依赖等问题。它们关注的角度不同，合在一起比单次“帮我看看有没有问题”可靠得多。

当然，浏览器上传环节也给我安排了节目。Google AI Studio 不支持直接上传部分 `.yml` 文件，浏览器自动上传还反复超时。艹了，最先进的模型面前，我居然卡在文件扩展名。

最后我修改工作流，让它自动生成 `02_ai_studio_upload`：把审核载荷逐字节复制为 `.txt`，保留原始载荷的 SHA256，用户只需全选上传。这样既绕开格式限制，也能证明 AI 看到的内容没有被悄悄改写。

## 连 WARNING 都不能数错

双审汇总器曾出现一个很典型的解析错误：一条 WARNING 下面有多个缩进子项，程序却把它们误算成五条 WARNING。看起来只是统计数字不漂亮，实际上会直接影响风险判断。

我修正了解析逻辑，并加入回归测试，防止同类格式再次把子项当成独立警告。最终完整回归验证通过：通用测试 95 项、矿井兼容测试 51 项、提交检查测试 14 项。数字不代表工作流永远不会错，但至少说明这次改造没有靠“我点了几遍，看起来没问题”收尾。

这些信息只用于项目说明、引用和维护，严格不会进入论文、图表、模型结果或匿名提交包。当前工作流已经备份到 GitHub 私有仓库，后续还会继续优化。

这次最大的经验是：AI 可以生成，也可以质疑，但它不能替代可追溯的证据，更不能替代人的责任。一个好的工作流，不是按钮特别多，也不是界面看起来像控制中心，而是在出错时，我能立刻知道哪里错了、谁来确认，以及怎样把当时的过程重新复现出来。
