这次我没有把目标简单设成“做出一道 C 题”。更准确地说，我想检验的是：在一台真实的 Windows 电脑上，能不能把一套数学建模任务从原始题目和附件开始，一直推进到数据审计、模型选择、论文编译、独立复现、匿名打包和最终审计，并且让这次踩过的坑，不再原样出现在下一道题里。

说白了，我不想再靠“这次刚好跑通”交差。

我使用 Codex 协助代码编写、运行诊断、文档整理和论文初稿，但没有把“程序跑完”当作正确性的同义词。每个关键结论都要能追溯到脚本输出；每次正式运行都要留下输入哈希、日志和产物清单；人工审批、比赛规则确认和结论责任，也没有交给 AI 冒充完成。

最后得到的不只是一篇论文，而是一条经过实战暴露问题、再逐项加固的建模流水线。

完整论文：[《NIPT 检测时点的分层优化与女胎异常判定》PDF（56 页）](/papers/cumcm2025-c-modeling-paper.pdf)

> 说明：这是我在 2026 年对 2025 历史赛题完成的技术候选成果，仅作为研究记录，不代表已经完成任何赛区、学校或正式参赛审批。

## 一、先建立边界，再开始建模

项目使用独立的 ASCII 标识 `CUMCM2025-C`，与以前的练习项目完全隔离。题目 PDF 和 Excel 附件先复制到只读输入区，并登记文件大小和 SHA256。这样做看起来有点“工程化”，但它解决了一个很现实的问题：后面如果结果发生变化，必须先回答究竟是代码变了、环境变了，还是输入文件被替换了。

正式建模前，我先完成了三件事：

1. 固定题目、附件和项目目录，禁止分析脚本修改原始文件；
2. 创建项目自己的 Python 虚拟环境，不依赖系统环境里“碰巧装过”的包；
3. 先做数据字典和异常审计，再决定模型与验证方式。

这一阶段很快证明了必要性。附件并不是可以直接扔进模型的干净表格：孕周同时出现 `20w` 和 `20w+1` 两种格式；检测日期混有 Excel 序列号和日期对象；男胎和女胎数据都存在同一母体的重复测量；同一母体、同一天、同一抽血次数还出现技术重复；女胎有空表头列、一个可重算的 BMI 缺失值，以及同一技术事件标签不一致的情况。

更关键的是，男胎 1082 行数据聚合后只有 1063 个独立检测事件、267 名孕妇。女胎则是 605 行、147 名孕妇。若按“行”随机切分训练集和测试集，同一名孕妇的数据就可能同时出现在两边，结果会因为信息泄漏而显得异常漂亮。因此，后续所有验证都按孕妇分组，纵向模型也明确处理母体内相关性。

## 二、四个问题怎么建模

这道题的难点不只是选一个算法，而是让四问共享一致的数据层级、验证边界和决策含义。

### 问题一：把重复测量当成重复测量

问题一研究男胎 Y 染色体浓度与孕周、BMI 等因素的关系。我选择以样条刻画孕周的非线性变化，并用随机截距混合模型处理同一孕妇的多次检测。由于残差存在尾部偏离和一定异方差，又补充了以孕妇为聚类单元、采用稳健协方差的 GEE 复核。

最终模型使用 1063 个检测事件、267 名孕妇。边际 R² 为 0.156，条件 R² 为 0.771，组内相关系数约为 0.729。这说明个体差异占据了相当大的变异，不能忽略母体层级。GEE 中孕周整体效应非常显著，BMI 系数为负，p = 0.044。不过，这些结果表达的是统计关联，不是因果关系。

### 问题二：把“分组”写成可复核的优化问题

问题二需要根据 BMI 分组并给出建议检测时点。我没有手工试几个区间，而是先拟合达到 Y 浓度阈值的概率，再将候选切点、建议周数、等待代价和组数惩罚放进动态规划。

最终得到 4 组建议：17.5 周、18.8 周、20.3 周和 25.0 周，平均遗憾约为 0.981 周。技术重复估计出的测量误差标准差约为 0.00471；测量误差模拟 200 次、母体自助法 200 次，最终均为 0 次失败。

![不同 BMI 个体的最早可靠检测周数与分组建议](/cumcm2025-c-q2-policy.webp)

*问题二：蓝点是个体最早可靠检测周数，红线是动态规划压缩后的分组建议。*

这里最重要的并不是四个数字，而是边界条件：最高 BMI 组仍有 2 名孕妇到 25 周没有达到模型设定的 90% 可靠度。因此，25 周只能解释为题设范围内的上限建议，不能包装成对极端个体的保证。

### 问题三：复杂模型没有赢，也要如实写

问题三加入年龄、身高、体重、孕次、产次和辅助生殖等因素。我采用带单调约束的梯度提升模型，并使用嵌套的母体分组验证，保证每条外层预测只出现一次、训练组和验证组没有母体重叠。

模型的 ROC-AUC 为 0.673，PR-AUC 为 0.922，Brier 分数为 0.109，10 组 ECE 为 0.027。但把预测结果转化成分组检测策略后，只保留了 2 组，平均策略遗憾达到 6.249 周，明显不如问题二的简单策略。

这是一次很有价值的“负结果”：更多变量和更复杂的模型并没有自动带来更好的决策。最终论文保留该模型作为多因素分析和对照，但没有宣称它优于问题二。对我来说，承认复杂模型没赢，比硬找一个角度把它写成“提升”更重要。

### 问题四：先搞清标签究竟代表什么

女胎数据中，AE 列“胎儿是否健康”全部为“是”，无法作为监督学习目标；真正有正负变化的是 AB 列的非整倍体检测标签，而且它也未必等同于确诊金标准。因此，问题四被严格限定为“预测附件中的 AB 检测标签”，不把它扩张成临床诊断模型。

最终选择弹性网逻辑回归。嵌套母体分组验证下，ROC-AUC 为 0.738，PR-AUC 为 0.411，灵敏度为 0.642，特异度为 0.725，平衡准确率为 0.683，Brier 分数为 0.0825。500 次母体自助法没有拟合失败。

![女胎异常标签候选模型的 ROC 与 PR 曲线](/cumcm2025-c-q4-roc-pr.webp)

*问题四：同一套外层分组验证下，弹性网模型在 ROC 与 PR 曲线上的表现。*

这些数字足以描述模型在当前附件上的区分能力，却远不足以支持临床应用。标签边界、样本量和技术重复冲突，都必须与性能指标一起出现。

## 三、真正危险的不是报错，而是“看起来成功”

整个过程中最值得记录的一次故障发生在问题二。第一轮 200+200 次重采样正常结束，程序退出码也是 0，但日志里出现了约 20 条 GEE 迭代上限警告。旧状态逻辑仍把这些拟合记为 `ok`。如果只看“命令是否成功”，这批结果会直接进入论文。

我把第一轮结果标记为无效，给拟合入口加入显式收敛检查、有限参数检查和阻断警告，再完整重跑。第二轮测量误差模拟与母体自助法才真正达到 200/200 全部收敛。

问题四也遇到类似情况：首轮弹性网产生了 29 个 `ConvergenceWarning`。尽管行为测试通过，模型仍不能进入正式结果。调整正则化候选、迭代上限和容差后，又在“任何收敛警告都视为错误”的条件下重跑，才完成封板。

真正让我后背发凉的，不是报错，而是程序一脸平静地告诉我“成功”。这也让我形成了一条以后不会再放松的规则：

> 退出码为零，只说明进程没有崩溃；它不证明统计模型收敛，更不证明结论可信。

## 四、Windows 环境暴露出的工具问题

模型之外，真正消耗时间的是环境和工具链。说实话，这部分有时比建模本身更折磨人。下面这些问题都在本次任务里真实发生过：

| 问题 | 表现 | 最终处理 |
|---|---|---|
| 工作目录错误 | 第一次补丁落到上级目录，差点形成两套项目树 | 项目改为原子初始化，并校验唯一绝对路径 |
| PowerShell 编码 | 从标准输入传入含中文的 Python 源码时，工作表名变成问号 | 中文脚本一律保存为 UTF-8 文件后执行 |
| Python 依赖漂移 | 系统解释器缺少科学计算、建模和测试依赖 | 每题独立 `.venv`，保存版本锁并执行 `pip check` |
| 持久任务启动失败 | 相对解释器没有按子进程工作目录解析 | 启动前把路径型可执行文件解析成绝对路径，并持久记录失败 |
| MiKTeX 挂起 | `kpsewhich` 查询缺失包时拉起安装器并等待 | 固定便携工具链、禁自动安装、单项 5 秒超时 |
| PDF 工具失效 | Poppler 实际位于 `Library/bin`，旧包装器指向不存在的 `bin` | doctor 自动探测真实位置并生成项目包装器 |
| 并行诊断丢结果 | 一个子命令非零退出导致整批结果不可见 | 独立检查改用 `allSettled`，逐项保存状态 |
| LaTeX 假错误 | 空模板启用参考文献，产生无引用错误和占位条目风险 | 默认关闭 bibliography，有真实引用后再启用 |
| 邮件交付不可用 | 会话没有可调用的发送接口 | 不虚报发送，改为交付真实本机文件并核对哈希 |

这些问题有一个共同点：它们不是“换个命令再试试”就算解决。若只修当前项目，下一题仍会重演。因此，修复被下沉到了公共工作流：项目初始化器、环境 doctor、持久任务系统、统计拟合防护、论文编译入口、PDF 预检、冻结完整性校验和匿名打包器。

## 五、从“能生成 ZIP”到“可以信任这个 ZIP”

早期打包器只检查运行状态是否写着 `frozen`。这并不够：冻结后的目录仍可能被修改，ZIP 也可能因为文件时间或清单顺序不同而每次产生不同哈希。

加固后的流程会在打包瞬间重新核验 76 个产物和冻结清单哈希；按固定顺序和时间戳写入；拒绝重复、大小写冲突、Windows 保留名和路径穿越；以 1 MiB 分块读取文件，并在写入时再次核对 SHA256 与大小；最后通过 CRC 自检，再原子替换正式 ZIP。

正式运行与独立复跑最终都被冻结。共比较 76 个输出：59 个文件字节完全一致，13 个 PDF 仅容器元数据不同但同名渲染图一致，4 个清单只含这些 PDF 哈希差异，实质结果 0 个不一致。匿名支撑包包含 160 个唯一条目，不含原始题目附件、本机路径或身份信息。

这套证据链把“我这里跑出来了”推进到了“另一次独立运行得到了同样的实质结果”。

## 六、论文生成也必须通过工程门禁

论文不是把图表粘进去就结束。第一次正式编译时遇到了缺少 DOI/URL 命令、长等宽标识符溢出版心、源码附录误包含生成器自身、Unicode 字符缺字，以及 AI 使用说明中的长编号越界等问题。

最终编译入口固定到项目本地工具链，预检同时检查：

- A4 页面和页数、文件大小；
- 未定义引用、缺字和行溢出；
- 匿名元数据与身份关键词；
- 每一页能否实际渲染；
- 论文引用的表格、图和指标是否来自冻结运行。

最终主论文为 56 页 A4，其中正文约 15 页，后附完整分析源码；AI 使用详情为 3 页 A4。这里的“通过”只代表技术预检通过，不代表已经完成正式参赛审批。

## 七、把这次踩坑变成下一题的起跑线

完成 C 题后，我没有直接停止，而是用最新模板创建了一个不含真实题目和数据的全新测试项目，模拟下一道题的冷启动。结果如下：

- 环境准备通过持久任务完成，约 2 分 12 秒，退出码为 0，stderr 为空；
- doctor 对项目路径、独立解释器、依赖、UTF-8、LaTeX、Perl、包装器和 Poppler 的检查全部通过；
- `pip check` 未发现损坏依赖；
- 空白论文干净编译为 3 页 A4，3/3 页面成功渲染；
- 通用回归测试、Python 语法扫描和 PowerShell 解析检查全部通过；
- 旧练习项目没有被运行或修改。

下一道题的标准启动顺序也因此变得很明确：

```powershell
python -m workflow_core.cli.init_project --project-id <ASCII_PROJECT_ID> `
  --competition "<比赛名称>" --problem "<题号>"

Set-Location competitions\<ASCII_PROJECT_ID>
.\scripts\bootstrap_environment.ps1
```

然后执行 doctor、登记输入哈希、完成数据审计与验证计划。只有这些检查没有错误，才允许启动正式建模任务。正式结果还要经过独立复跑、冻结完整性核验和匿名打包。

## 八、这次最重要的八条经验

1. **数据层级先于算法。** 有重复测量时，验证单位必须与真实独立主体一致。
2. **先做可解释基线。** 复杂模型没有赢过简单策略时，应保留负结果，而不是换一种口径宣布胜利。
3. **警告也是结果。** 收敛警告、资源警告和编译警告必须分级处理，关键警告直接阻断正式产物。
4. **不要依赖系统环境。** Python、MiKTeX、Perl、Poppler 都要在项目启动时明确检查。
5. **长任务要有稳定身份。** UI 中的一次执行句柄会丢，job ID、PID、状态文件和日志不会。
6. **冻结不等于批准。** `frozen` 只表示计算产物不再变化，H1-H8 等人工门禁仍需真人完成。
7. **规则不知道就是不知道。** 本次能核对到 2025 提交说明和 AI 规定，但没有用后续的页数、大小规则冒充 2025 规则。
8. **修复必须可回归。** 每个通用故障都先用失败测试复现，再修改公共组件，最后用全新项目冷启动证明修复有效。

## 结语

这次 C 题最有价值的成果，不是某个 AUC，也不是某个最优检测周数，而是把一次模型求解变成了可以检查、复现和追责的过程。

数学建模当然需要公式、算法和图表，但真正高质量的作品还需要回答更多问题：数据有没有泄漏？模型是否收敛？复杂方案是否真的优于基线？论文中的数字来自哪次运行？附件被改过吗？换一台环境还能复现吗？匿名包里有没有不该出现的内容？哪些结论仍然需要真人确认？

当这些问题都能用证据回答时，一次比赛练习才真正沉淀成下一次可以复用的能力。
