最近，我对自己的数学建模项目工作流进行了一次系统性升级。

最开始，我的目标很直接：让 **VS Code、MiKTeX、LaTeX Workshop、Python、MATLAB、Codex、Git 和 GitHub** 能够协同工作，完成数据处理、模型计算、图表生成、论文编写、版本管理和提交检查。

但随着项目逐渐完善，我发现一件很现实的事：

> 代码能运行、图表能生成、论文能编译，只能证明工具链基本可用，不能证明最终结论可靠。

一套真正能支撑国赛、校赛和长期积累的工作流，还要回答更多问题：

- 每个小问究竟要解决什么决策或科学问题？
- 模型选择有没有明确依据？
- 数据经历了哪些清洗、转换和异常处理？
- 论文中的数字能否追溯到具体代码、数据和运行记录？
- “准确”“稳定”“鲁棒”“最优”等结论有没有足够证据？
- 图表是否和原始结果一致，并经过人工复核？
- AI 生成的模型、代码和文字是否经过人工确认？
- 换一台电脑后，能否复现相同或可以解释的结果？
- 最终提交包是否混入身份信息、绝对路径、缓存或私有资料？

因此，这次升级的重点不再只是“把程序跑通”，而是建立一套覆盖建模全过程的质量闭环。

## 一、把通用工作流和具体赛题分开

以前，脚本、规则和检查逻辑很容易全部堆在一道题里。题目一换，很多东西就要重新搭建。

这次我把系统拆成了两层。

第一层是可以复用的通用能力：

- 竞赛规则与项目配置；
- 任务调度与运行追踪；
- 数据审计与配置校验；
- 模型插件与案例适配；
- 人工审核与批准流程；
- 证据链和图表管理；
- 论文质量检查；
- 匿名提交包生成；
- 跨题型回归测试。

第二层是具体比赛项目：

- 原始题目和附件；
- 原始数据与中间数据；
- 模型代码和参数配置；
- 运行结果与论文图表；
- 论文源文件；
- 支撑材料与最终提交文件。

现在的结构可以概括为：

```plaintext
workflow_core/sdk <- plugins <- cases / competitions
        ^
competition_profiles、paper_templates 作为受验证的数据输入
```

`workflow_core` 负责调度、Schema、运行记录、审批、证据和打包；`plugins` 负责题型能力；`cases` 负责把具体案例接入通用流程；`competitions` 则保留每次比赛自己的数据、代码和论文。

Python 是目前唯一的总调度层。它统一启动数据处理、模型计算、案例程序和质量检查，并记录每一步的输入与输出。MATLAB 适配器仍在后续计划中，目前还没有把“计划支持”写成“已经完成”。

## 二、先设计论证，再开始写代码

这次新增了 `question_storyboard.yml`，我把它理解为每个小问的“建模作战卡”。

在真正编码之前，它要求先写清楚：

- 这个小问要解决什么；
- 输入数据和必须交付的结果是什么；
- 约束条件和核心难点在哪里；
- 有哪些候选模型；
- 为什么最终选择这个模型；
- 关键假设和算法前提是什么；
- 准备怎样验证和做敏感性分析；
- 需要哪些图表；
- 论文对应哪个章节；
- 最终结论能说到什么程度，又不能说到什么程度。

过去很容易拿到题目后马上处理数据、写代码，再根据结果反向组织论文。这样看起来很快，但也容易出现几个问题：模型和题目目标脱节，不同小问之间没有衔接，结果生成了很多，却不知道究竟要证明什么。

逐问故事板把顺序调整成：

> 问题定义 → 候选方案 → 选择依据 → 实现 → 验证 → 论文表达。

它并不能替我做建模判断，但会迫使我在动手前先把论证路线想清楚。

## 三、给数据和假设建立明确边界

数学建模里的很多错误并不是模型造成的，而是数据含义、单位、字段或预处理规则在不同脚本之间悄悄发生了变化。

因此，我开始给关键数据建立数据契约，记录：

- 数据文件和来源；
- 字段名称、类型与单位；
- 缺失值和异常值的处理规则；
- 坐标系、时区或时间基准；
- 清洗脚本和数据版本；
- 文件 SHA256；
- 是否允许进入匿名提交包。

例如，同一个“距离”必须说明是米、千米还是归一化距离；时间字段要明确采样间隔与起止边界；空间数据则必须说明坐标系。

假设也不再只是论文开头的一组固定句子。`assumptions_registry.yml` 会把题面条件和作者假设分开，并记录每项假设使用在哪个小问、为什么提出、失效后会影响什么、怎样验证、是否需要敏感性分析。

这样做之后，假设不再是写作形式，而真正成为模型有效性的一部分。

## 四、让每一次计算都有唯一身份

为了实现可复现，每次正式计算都会生成唯一的 `run_id`。

一次运行至少记录：

- 项目、问题和模型名称；
- 运行时间、Git commit 与当前分支；
- 数据文件哈希和参数配置；
- 随机种子；
- Python、依赖库和操作系统信息；
- 输入、输出与日志位置；
- 运行状态和审核状态。

运行结果被隔离在：

```plaintext
05_model_results/runs/<run_id>/
```

它的生命周期是：

```plaintext
created -> running -> waiting_approval / paused
        -> completed -> frozen
        -> failed / invalidated / tampered
```

完成后的输出不能随意覆盖。完整输出清单或文件哈希发生变化时，运行会被标记为被篡改；配置变化后也不能继续冒充旧运行，而要创建新的 `run_id`，并记录它和上一轮运行的关系。

这样，当论文里出现一个指标时，我可以继续追问：

> 它来自哪次运行？用了哪版数据？采用了什么参数？对应哪个代码版本？是否经过验证和人工批准？

如果模型包含随机初始化、启发式算法、蒙特卡洛模拟或机器学习训练，还需要固定随机种子，并在必要时执行多次实验，报告均值、标准差、置信区间或结果分布，而不是只留下表现最好的一次。

## 五、把论文结论和真实证据连起来

这次工作流把四类文件连接到了一起：

```plaintext
claims.csv
evidence_links.csv
question_result_cards.csv
abstract_matrix.csv
```

`claims.csv` 登记论文中的重要主张，`evidence_links.csv` 把主张连接到真实的运行结果、指标、图表和文件哈希。

小问结果卡只保留每个小问真正需要进入论文的核心结果，包括模型、主要指标、数值、单位、验证证据、适用条件、局限和人工审核状态。

摘要矩阵则要求每个小问都对应研究目标、方法、结果、验证和局限。摘要中的数字必须来自已经冻结并验证的证据，不能靠记忆转抄，更不能为了让文字好看临时补一个结果。

整个填写顺序被固定为：

1. 完成并冻结 analysis run；
2. 登记 claims 和 evidence links；
3. 填写逐问结果卡；
4. 最后压缩成摘要矩阵。

目前，结果卡和摘要矩阵仍然保持空表，只完成了结构、Schema 和检查逻辑。它们要等真实 analysis run 冻结后，再由队员填写和复核。空表不是失败，而是明确表示“现在还没有可以进入正式论文的结果”。

对于强结论，证据要求也会更严格：

- “准确”需要误差指标、测试集结果或交叉验证；
- “稳定”需要重复实验、扰动实验或方差分析；
- “鲁棒”需要参数、数据或场景扰动；
- “最优”需要目标函数、约束、可行性和最优性依据；
- “优于其他模型”需要统一数据、统一指标和公平对照。

证据链的目的不是把论文写得复杂，而是避免“结论很强，证据很弱”。

## 六、不同题型使用不同的验证方式

我以前也容易把“验证模型”理解成一个固定动作。但不同题型真正需要证明的事情并不一样。

现在，通用核心只定义验证类别，具体方法交给题型插件：

| 题型 | 重点验证内容 |
| --- | --- |
| 机制 / 物理模型 | 量纲、守恒、参数物理意义、边界与极端情况 |
| 网络 / 路径模型 | 图结构、路径可行性、边权前提、后备算法与替代结构 |
| 空间数值模型 | 网格、时间步长、稳定性、收敛性与边界处理 |
| 时间序列模型 | 时间切分、滚动回测、数据泄漏、残差和朴素基线 |
| 评价 / 排序模型 | 指标方向、权重来源、替代权重与排名稳定性 |
| 优化模型 | 目标与约束、可行解、最优间隙、多初值和局部最优风险 |
| 启发式 / 仿真 | 随机种子、重复实验、置信区间、收敛趋势和失败场景 |

H4 审核不只要说明“做了哪些验证”，还要说明“为什么选择这些验证”，以及为什么没有选择其他验证。

当前第一轮只接入了少量实验性插件，时间序列等题型还没有实现。我更愿意让系统明确说“暂不支持”，而不是输出一个看起来完整、实际没有经过验证的结果。

## 七、AI 可以参与检查，但不能批准自己

此前，程序输出、AI 意见、人工检查和修订记录分散在不同目录里，很难快速判断项目进行到了哪一步。

现在，工作流会在 H1 到 H8 每个阶段自动生成审核包：

| 门禁 | 审核内容 |
| --- | --- |
| H1 | 题目理解与逐问叙事 |
| H2 | 数据审计与预处理 |
| H3 | 模型选择与假设 |
| H4 | 验证计划与审计预算 |
| H5 | 基线结果与基本不变量 |
| H6 | 替代模型与结构比较 |
| H7 | 论文、证据与图表 |
| H8 | 最终提交与双人复核 |

每个审核包包含需要复核的文件副本、文件哈希、AI 提示词、AI 完整回复位置、人工清单、人工结论和批准命令。

```plaintext
09_review_packages/<run_id>/
├── 00_REVIEW_INDEX.csv
└── 01_H1_problem_definition_<hash>/
    ├── 01_review_files/
    ├── 02_AI_PROMPT.md
    ├── 03_AI_REVIEW.md
    ├── 04_HUMAN_CHECKLIST.md
    ├── 05_HUMAN_DECISION.md
    └── 06_APPROVE_COMMAND.txt
```

Codex 和其他 AI 工具可以：

- 生成代码初稿和补充测试；
- 检查变量、单位和潜在逻辑冲突；
- 对比正文、结果文件与图表；
- 整理日志和生成审核清单；
- 辅助修改 LaTeX；
- 扫描提交包风险。

但 AI 不能批准 H1-H8，也不能替队员确认题目理解、模型选择、关键假设、异常数据处理、主结果、强结论和最终摘要。

H8 还要求两名不同队员分别完成数值模型终审和论文合规终审。自动化可以把问题暴露出来，但关键判断仍然要由人承担。

## 八、图表也必须成为证据的一部分

图表不是程序生成后直接插进论文就结束了。

正式图表需要在 `figure_manifest.csv` 中登记用途、来源数据、支持的结论、来源运行、坐标轴、单位、分辨率和人工复核状态。

复核时至少要确认：

- 横纵坐标、单位和图例是否完整；
- 是否存在截断坐标轴造成的误导；
- 图中数值能否在结果文件中找到；
- 图片与正文描述是否一致；
- 黑白打印后是否仍然能区分；
- 是否包含用户名、绝对路径或隐私信息；
- 分辨率与论文宽度是否适合打印。

表格也要检查数值精度、单位、合计、排序和正文引用是否一致。程序结果正确但论文展示错误，同样会让整个论证失效。

## 九、把问题挡在提交之前

为了避免所有问题都积累到最后一天，工作流把检查拆成多个层级：

- **数据门禁**：字段、类型、缺失值、单位、哈希和审核状态；
- **模型门禁**：候选模型、选择依据、假设、参数来源和验证计划；
- **运行门禁**：执行状态、日志、随机种子、结构化结果和警告；
- **结果门禁**：是否回答题目、是否有基准、误差和敏感性分析；
- **论文门禁**：摘要、正文、结论、图表、公式、引用和占位符；
- **提交门禁**：身份信息、绝对路径、私有数据、缓存、PDF 和压缩包结构。

提交材料采用 allowlist 正向打包：只有明确登记为允许提交的文件才能进入最终包。优秀论文、历史项目、内部笔记和本机配置只作为私有参考，不会因为“忘了排除”而混进 Git 或匿名提交材料。

提交前还会扫描姓名、学校、学号、邮箱、用户目录、电脑用户名、Git 远程地址、文件元数据、隐藏文件和私有参考资料。

这不仅是隐私问题，也关系到匿名评审和比赛合规。

## 十、没有推倒旧流程，而是把它变成回归案例

这次升级没有重写已经跑通的矿井突水 Q1—Q4 练习流程。

原来的兼容入口仍然保留：

```powershell
.\.venv\Scripts\python.exe 04_code\run_all.py
```

旧流程继续负责数据检查、清洗、Q1—Q4、验证、灵敏度分析、图表和导出。通用流程通过案例适配器启动它，并把输出复制到独立运行目录。

这样做有两个好处：一是不会为了“架构更漂亮”破坏已经验证过的程序；二是每次升级通用核心后，都可以用旧题检查运行、LaTeX、证据和提交检查是否出现回归。

目前已经落地的是第一轮最小闭环，包括：

- 配置 Schema 与资产解析；
- 唯一 `run_id` 和不可变输出；
- H1—H8 人工门禁与审核包；
- 逐问故事板、结果卡和摘要矩阵；
- 插件验证梯度；
- 证据与图表门禁；
- allowlist 匿名打包；
- 冷启动复现和矿井案例桥接。

尚未完成的部分也很明确：MATLAB 适配器、Word 模板适配、时间序列插件及其案例，以及更完整的图表故事板和反方复审工具。这些会继续做，但不会提前出现在“已完成”列表里。

## 十一、这次升级带给我的改变

最初搭建这套工作流时，我更关心的是工具是否齐全：编辑器能不能写、环境能不能跑、论文能不能编译、GitHub 能不能同步。

现在我更关心的是另一组问题：

- 这个结果为什么值得相信？
- 如果换一个模型，主要决策会不会改变？
- 一个结论的边界在哪里？
- 队友能不能在不问我的情况下找到来源？
- 比赛结束很久以后，我还能不能重新理解当时做了什么？

这让我越来越觉得，平台不会替代能力，但会决定能力能不能被稳定地发挥出来。

一个人可能会写模型、会调参数、会画图，但如果数据版本混乱、结果无法追溯、论文数字靠手工转抄，那么这些能力很容易在最后一步被消耗掉。相反，一套可靠的工作流可以把低级错误和重复劳动提前挡住，让人把时间放在真正重要的建模判断上。

Codex 在这里更像协作工程师：它帮助实现、检查和整理；Git 记录代码与配置变化；运行追踪系统保存计算过程；证据链连接结果与论文；最后的判断和批准仍然由人完成。

## 总结

这次升级最大的变化，是把数学建模工作流从：

> 代码能运行、图表能生成、论文能编译

进一步推进为：

> 问题有拆解、模型有依据、数据有版本、运行有记录、结果可追溯、结论有证据、图表有复核、AI 有审核、提交有门禁。

它的目标不是让 AI 自动完成一场数学建模比赛，也不是用复杂的工程结构替代建模思考。

我真正想解决的是：在时间紧张、任务复杂、多人协作、文件数量快速增长的环境里，尽可能减少低级错误和无法追踪的问题，把有限的精力留给题目理解、模型选择、结果解释和论文表达。

这套工作流还没有完成。接下来，我会继续扩展时间序列、评价模型、整数规划、多目标优化、机器学习、空间分析和随机模拟等题型，并用更多不同类型的题目检验它是否真的通用。

但至少现在，它已经不再只是一个“能跑的项目”，而开始变成一套可以复现、可以审查、也敢于承认尚未完成部分的数学建模工作流。
